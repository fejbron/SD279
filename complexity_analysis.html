Motivation

Why Analyse Algorithms?

Algorithm analysis predicts how a program behaves before writing a single line of code — saving engineering time and preventing disasters at scale.

  • Two correct algorithms may differ by 1000× in speed on large inputs
  • Hardware improvements (Moore's Law) slow down — algorithmic efficiency lasts forever
  • Understand scaling behaviour: what happens when n doubles, triples, or reaches billions?
  • Compare algorithms independently of hardware, language, or implementation
  • Essential for technical interviews, system design, and production engineering
Hardware vs Algorithm

A 10× faster computer only helps if the algorithm is the same. Going from O(n²) to O(n log n) on n=10⁶ is a 50,000× improvement — no hardware can match that.

Example

Bubble Sort vs Merge Sort on n=1,000,000:
Bubble: ~10¹² ops · Merge: ~20×10⁶ ops
That's 50,000× difference.

We analyse growth rate, not exact counts — constants and lower-order terms are irrelevant at scale.

Motivation

How Do We Measure Running Time?

Empirical / Experimental

Run the program, measure wall-clock time. Problem: depends on hardware, OS, compiler optimisations, programming language, other processes running.

Analytical / Theoretical

Count the number of primitive operations (comparisons, assignments, arithmetic) as a function of input size n. Machine-independent.

We use theoretical analysis — express running time as T(n), a function of input size n.

Primitive Operations

Evaluating an expression, assigning a value, indexing an array, calling/returning from a function, following a pointer, comparison — each counts as O(1).

Input Size n

Depends on the problem: array length, number of characters in a string, number of vertices + edges in a graph, number of bits in an integer.

# T(n) for this loop?
total = 0 # 1 op
for i in range(n): # n iterations
  total += i # 1 op per iter
# T(n) = 1 + 2n → O(n)

Types of Analysis

Best, Average & Worst Case

Best Case — Ω(f(n))

The minimum number of operations over all possible inputs of size n. Usually not very informative — easy inputs don't reflect real-world behaviour.

Linear Search: target is arr[0] → O(1)

Average Case — Θ(f(n))

Expected number of operations over all inputs assuming a probability distribution (usually uniform random). Most realistic but hardest to compute.

Linear Search: target at position n/2 → O(n/2) = O(n)

Worst Case — O(f(n))

The maximum number of operations over all possible inputs. Gives a guaranteed upper bound. Most commonly used in practice — guarantees performance.

Linear Search: target not in arr → O(n)

Unless stated otherwise, Big-O always refers to the worst-case — it's the only guarantee you can give to users of your algorithm.

Foundations

Rate of Growth

The rate of growth of a function describes how fast its value increases as n→∞. We drop constants and lower-order terms — only the dominant term matters.

  • T(n) = 5n² + 3n + 100 → dominant: n²
  • At n=1000: 5×10⁶ vs 3000 vs 100 — the n² term is ~1667× larger than 3n
  • We say T(n) grows like n² — ignore coefficients and lower terms
  • This simplification is valid because as n→∞, lower-order terms become negligible
T(n) = 5n² + 3n + 100
→ drop 3n, 100 (lower order)
→ O(n²)
T(n) = 100n + 50n log n
→ n log n dominates n
→ O(n log n)
T(n) = 3·2ⁿ + 1000n²
→ 2ⁿ dominates n²
→ O(2ⁿ)

Big-O Notation

Big-O — Formal Definition

f(n) = O(g(n))
if ∃ constants c > 0 and n₀ ≥ 1 such that
f(n) ≤ c·g(n)  ∀ n ≥ n₀

Big-O gives an asymptotic upper bound — g(n) grows at least as fast as f(n) beyond some threshold n₀.

  • Read: "f(n) is Big-O of g(n)"
  • "f(n) grows no faster than g(n)"
  • The constant c absorbs any fixed multiplicative overhead
  • We only care about behaviour as n → ∞

Prove: 3n + 5 = O(n)

# Need: 3n + 5 ≤ c·n for all n ≥ n₀
# Choose c = 4, n₀ = 5:
# n=5: 15+5=20 ≤ 4·5=20 ✓
# n=10: 30+5=35 ≤ 4·10=40 ✓
# n=100: 305 ≤ 400 ✓
# Holds for all n ≥ 5 with c=4
# Therefore 3n+5 = O(n) □
Key insight

Big-O is not unique. If f(n)=O(n), then also f(n)=O(n²), O(n³), etc. We always state the tightest (smallest) correct bound.

08 / 40
Big-O Notation

Big-O — Worked Examples

Example 1: T(n) = 5n² + 3n + 1

Dominant term: n². Claim: T(n) = O(n²).
Proof: 5n²+3n+1 ≤ 5n²+3n²+n² = 9n² for n≥1.
Choose c=9, n₀=1. ✓

Example 2: T(n) = 2ⁿ + n¹⁰⁰

Exponential dominates polynomial. T(n) = O(2ⁿ).
Since n¹⁰⁰ ≤ 2ⁿ for n ≥ 1000, we have T(n) ≤ 2·2ⁿ. Choose c=2, n₀=1000. ✓

Example 3: T(n) = log₂(n³)

log₂(n³) = 3·log₂(n) = O(log n).
Logarithm base changes are just constant multiples — all log bases are equivalent in Big-O.

Example 4: T(n) = n·log n + n

n·log n dominates n (for n≥2, log n≥1 so n·log n ≥ n).
T(n) = O(n log n).

Example 5: T(n) = sum from i=1 to n of i

Σi = n(n+1)/2 = n²/2 + n/2 = O(n²).
This is why nested loops (outer n, inner up to i) give O(n²).

The step-by-step rule: expand → identify dominant term → state O(dominant term).

Big-Omega (Ω) Notation

Big-Ω — Lower Bound

f(n) = Ω(g(n))
if ∃ constants c > 0 and n₀ ≥ 1 such that
f(n) ≥ c·g(n)  ∀ n ≥ n₀

Ω gives an asymptotic lower bound — f(n) grows at least as fast as g(n) beyond n₀.

  • "f(n) is Big-Omega of g(n)"
  • "f(n) grows at least as fast as g(n)"
  • Used to prove lower bounds on algorithm complexity
  • E.g. any comparison-based sort requires Ω(n log n) — proved via decision tree

Prove: 3n + 5 = Ω(n)

# Need: 3n+5 ≥ c·n for all n ≥ n₀
# Choose c = 1, n₀ = 1:
# 3n+5 ≥ 1·n for all n ≥ 1 ✓
# (since 3n+5 ≥ n always)
# Therefore 3n+5 = Ω(n) □
Lower bound uses

Proving that an algorithm cannot be faster than Ω(f(n)). E.g., you must read all n inputs → any algorithm is Ω(n). Sorting by comparisons → Ω(n log n).

Big-Theta (Θ) Notation

Big-Θ — Tight Bound

f(n) = Θ(g(n))
if ∃ constants c₁, c₂ > 0 and n₀ ≥ 1 such that
c₁·g(n) ≤ f(n) ≤ c₂·g(n)  ∀ n ≥ n₀

Θ is a tight asymptotic bound — g(n) and f(n) grow at exactly the same rate. f(n) = Θ(g(n)) iff f(n) = O(g(n)) AND f(n) = Ω(g(n)).

  • "f(n) is Theta of g(n)"
  • "f(n) and g(n) are asymptotically equivalent"
  • Strongest of the three bounds — exact growth rate
  • Not always achievable — best and worst case may differ

Prove: n(n+1)/2 = Θ(n²)

# n(n+1)/2 = n²/2 + n/2
# Upper: n²/2 + n/2 ≤ n² for n≥1
# → O(n²) with c₂=1, n₀=1 ✓

# Lower: n²/2 + n/2 ≥ n²/2 for n≥1
# → Ω(n²) with c₁=1/2, n₀=1 ✓

# Both hold → Θ(n²) □
When Θ exists

Θ(f(n)) exists only when best and worst cases have the same growth rate. Merge Sort is always Θ(n log n). Quick Sort is O(n log n) but NOT Θ(n log n) — worst case is O(n²).

Little-o and Little-ω

Little-o and Little-ω

Little-o and little-ω are strict versions — the bound does not touch. They describe algorithms that are strictly slower or faster, not just bounded.

f(n) = o(g(n))
if for ALL c > 0, ∃ n₀ such that
f(n) < c·g(n)  ∀ n ≥ n₀
Equivalently: lim(n→∞) f(n)/g(n) = 0
f(n) = ω(g(n))
if for ALL c > 0, ∃ n₀ such that
f(n) > c·g(n)  ∀ n ≥ n₀
Equivalently: lim(n→∞) f(n)/g(n) = ∞
NotationMeaningAnalogy
O(g)f ≤ c·g (asymp)f ≤ g
Ω(g)f ≥ c·g (asymp)f ≥ g
Θ(g)c₁g ≤ f ≤ c₂gf = g
o(g)f/g → 0f < g (strict)
ω(g)f/g → ∞f > g (strict)
Example

n = o(n²): lim(n→∞) n/n² = lim 1/n = 0 ✓
n² = ω(n): lim n²/n = lim n = ∞ ✓
n = O(n) but n ≠ o(n) (limit = 1, not 0)

Properties

Properties of Asymptotic Notation

Reflexivity

f(n) = O(f(n))  ·  f(n) = Ω(f(n))  ·  f(n) = Θ(f(n))

Transitivity

If f = O(g) and g = O(h) → f = O(h)
Same holds for Ω, Θ, o, ω

Symmetry (Θ only)

f(n) = Θ(g(n)) ↔ g(n) = Θ(f(n))

Transpose Symmetry

f=O(g) ↔ g=Ω(f)
f=o(g) ↔ g=ω(f)

Sum Rule

O(f) + O(g) = O(max(f,g))
e.g., O(n²) + O(n) = O(n²)

Product Rule

O(f) × O(g) = O(f×g)
e.g., O(n) × O(log n) = O(n log n)

Constant Factors

O(c·f) = O(f) for any constant c > 0
e.g., O(3n²) = O(n²)

These rules let you simplify T(n) expressions quickly without full formal proofs.

Complexity Classes

O(1) — Constant Time

Running time does not depend on n at all. The same number of operations regardless of input size — the holy grail of algorithms.

  • Array index access: arr[i]
  • Hash map lookup (average case)
  • Stack push/pop
  • Linked list head insert/delete
  • Arithmetic operations
# All O(1) — no loop, no recursion
def get_first(arr):
  return arr[0] # O(1)

def swap(arr, i, j):
  arr[i], arr[j] = arr[j], arr[i]
  # O(1) — fixed ops

def is_even(n):
  return n % 2 == 0 # O(1)
T(n) = c    (any constant)
Graph: flat horizontal line
16 / 40
Complexity Classes

O(log n) — Logarithmic Time

The algorithm halves (or reduces by a constant fraction) the problem size at each step. Incredibly efficient — log₂(10⁹) ≈ 30.

  • Binary search on sorted array
  • BST operations (balanced)
  • Heap insert/extract
  • Integer multiplication with repeated squaring
  • Finding an element in a balanced BST
# Binary search — O(log n)
def binary_search(arr, t):
  lo, hi = 0, len(arr)-1
  while lo <= hi: # log₂n iters
    mid = (lo+hi)//2
    if arr[mid] == t: return mid
    elif arr[mid]<t: lo=mid+1
    else: hi=mid-1
Why log n?

Each iteration: n → n/2 → n/4 → … → 1. After k steps: n/2ᵏ = 1 → k = log₂(n). So at most ⌊log₂n⌋ + 1 iterations.

17 / 40
Complexity Classes

O(n) — Linear Time

Running time grows proportionally to input size. Must look at each element at least once. Often optimal for problems that require reading the entire input.

  • Linear search
  • Finding max/min in unsorted array
  • Counting occurrences
  • Traversing a linked list
  • Counting sort (for fixed range)
# O(n) — single loop
def find_max(arr): # n ops
  best = arr[0]
  for x in arr[1:]:
    if x > best: best = x
  return best

# Two sequential O(n) loops → O(n)
def sum_and_max(arr):
  total = sum(arr) # O(n)
  mx = max(arr) # O(n)
  return total, mx # O(n)+O(n)=O(n)
Sequential loops

Two separate O(n) loops = O(2n) = O(n). Three loops = O(3n) = O(n). Constants are dropped.

18 / 40
Complexity Classes

O(n log n) — Linearithmic Time

The most common complexity for divide-and-conquer algorithms. Proven optimal for comparison-based sorting — no algorithm can do better.

  • Merge Sort, Heap Sort, Quick Sort (avg)
  • FFT (Fast Fourier Transform)
  • Building a balanced BST from sorted data
  • Many divide-and-conquer algorithms
Intuition

n log n = n (work per level) × log n (number of levels in divide-and-conquer). Think Merge Sort: n elements, split log n times.

# Merge Sort — O(n log n)
def merge_sort(arr):
  if len(arr) <= 1: return arr
  mid = len(arr) // 2
  # log n levels of recursion
  L = merge_sort(arr[:mid])
  R = merge_sort(arr[mid:])
  return merge(L, R) # O(n) each

# T(n) = 2T(n/2) + O(n)
# By Master Theorem → O(n log n)

Complexity Classes

O(n²) — Quadratic Time

Typically arises from nested loops where each loop runs n times. Acceptable for n ≤ 10,000; unacceptable for n ≥ 10⁶.

  • Bubble Sort, Selection Sort, Insertion Sort
  • Comparing all pairs of elements
  • Naive matrix addition O(n²) for n×n matrix
  • Floyd-Warshall shortest path O(V²) (actually O(V³))
# Nested loop → O(n²)
def bubble_sort(arr):
  n = len(arr)
  for i in range(n): # n times
    for j in range(n-i-1): # n-i times
      if arr[j]>arr[j+1]:
        arr[j],arr[j+1]=arr[j+1],arr[j]
# Total: n + (n-1) + … + 1 = n(n-1)/2 = O(n²)
Not always bad

Insertion sort is O(n²) worst-case but O(n) best-case (sorted input). Used inside TimSort for small subarrays because cache locality outweighs the complexity.

Complexity Classes

O(n³), O(2ⁿ), O(n!) — Dangerous Complexities

O(n³) — Cubic

Three nested loops. Feasible for n ≤ 500.

# Matrix multiplication
for i in range(n):
 for j in range(n):
  for k in range(n):
   C[i][j]+=A[i][k]*B[k][j]

n=1000 → 10⁹ ops ~ 1 second

O(2ⁿ) — Exponential

Each new element doubles the work. Only feasible for n ≤ 20–25.

# Fibonacci (naive)
def fib(n):
 if n<=1: return n
 return fib(n-1)+fib(n-2)

n=50 → ~10¹⁵ ops (years!)

O(n!) — Factorial

All permutations. Only feasible for n ≤ 12–13.

# Travelling Salesman
# (brute force)
# Try all n! routes
# n=20 → 2.4×10¹⁸ ops

n=20 → longer than Universe age

O(2ⁿ) and O(n!) algorithms are practically infeasible for large n. Dynamic programming often reduces O(2ⁿ) to O(n²) or O(n³).

Code Analysis

Analysing Single Loops

Loop runs n times → O(n)
for i in range(n): # n iters
  print(i) # O(1) body
# Total: O(n)
Loop runs n/2 times → O(n)
i = 0
while i < n: # n/2 iters
  process(i)
  i += 2 # step by 2
# Total: O(n/2) = O(n)
Loop halves i → O(log n)
i = n
while i > 1: # log₂n iters
  process(i)
  i //= 2 # halve each time
# Total: O(log n)
Loop multiplies i → O(log n)
i = 1
while i < n: # log₂n iters
  process(i)
  i *= 2 # double each time
# i: 1, 2, 4, 8, … → O(log n)
Loop runs √n times → O(√n)
i = 1
while i*i <= n: # √n iters
  process(i)
  i += 1
# Total: O(√n)

Rule: count loop iterations as a function of n. Step size and condition determine the complexity.

Analysing Nested Loops

Independent nested loops → O(n²)
for i in range(n): # n
  for j in range(n): # × n
    process(i, j) # O(1)
# T = n × n × O(1) = O(n²)
j depends on i → O(n²)
for i in range(n): # n
  for j in range(i): # 0,1,…,i
    process(i, j)
# T = 0+1+2+…+(n-1) = n(n-1)/2 = O(n²)
Three nested loops → O(n³)
for i in range(n): # n
 for j in range(n): # × n
  for k in range(n): # × n
    process(i,j,k)
# T = n³ = O(n³)
Outer O(n), inner O(log n) → O(n log n)
for i in range(n): # n
  j = n
  while j > 1: # log n
    process(i, j)
    j //= 2
# T = n × log n = O(n log n)

Space Complexity

Space Complexity

Space complexity = total memory used by an algorithm as a function of n. Includes input, output, auxiliary data structures, and the call stack.

Total Space

All memory used — including the input array itself. Usually O(n) minimum just to store the input.

Auxiliary Space

Extra memory beyond the input. In-place algorithms use O(1) auxiliary space. Merge Sort uses O(n) auxiliary (the merge buffer).

Interviews usually ask for auxiliary space. In-place algorithms (O(1) extra) are preferred when memory is constrained.

# O(1) auxiliary — in-place swap
def reverse(arr):
  lo, hi = 0, len(arr)-1
  while lo < hi:
    arr[lo],arr[hi]=arr[hi],arr[lo]
    lo+=1; hi-=1
# Space: O(1) — only lo,hi variables

# O(n) auxiliary — new array
def reverse_copy(arr):
  return arr[::-1] # O(n) copy

# Merge Sort: O(n) aux
# Quick Sort: O(log n) aux (stack)
# DFS on tree: O(h) aux (h = height)
Call stack space

Recursive algorithms use O(depth) stack frames. Recursive DFS on a tree of height h uses O(h) = O(log n) for balanced, O(n) for skewed.

Space Complexity

Time-Space Trade-Off

You can often trade space for time — use extra memory (memoisation, lookup tables) to avoid recomputation, achieving faster runtime.

Memoisation (DP)

Fibonacci naive: O(2ⁿ) time, O(n) stack.
Fibonacci memoised: O(n) time, O(n) space.
Trade O(n) space to gain from O(2ⁿ) → O(n).

Lookup Tables

Precompute trig values: O(n) space, O(1) lookup instead of O(1) compute per call (with expensive constants). Hash maps cache expensive computations.

Counting Sort

Uses O(k) extra space (k = value range) but achieves O(n+k) time — faster than O(n log n) comparison sorts when k is small.

There is no universally "best" algorithm — the right choice depends on your time and space constraints.

Recurrence Relations

What Is a Recurrence Relation?

A recurrence relation expresses T(n) in terms of T on smaller inputs — naturally arising from recursive algorithms.

  • T(n) = time to solve problem of size n
  • Base case: T(1) = c (some constant)
  • Recursive case: T(n) in terms of T(n/2), T(n-1), etc.
  • Goal: find a closed-form (non-recursive) expression for T(n)
Binary Search
T(n) = T(n/2) + O(1)
T(1) = O(1)
→ T(n) = O(log n)
Merge Sort
T(n) = 2T(n/2) + O(n)
T(1) = O(1)
→ T(n) = O(n log n)
Factorial
T(n) = T(n-1) + O(1)
T(0) = O(1)
→ T(n) = O(n)

Recurrence — Substitution

Solving Recurrences — Substitution Method

Guess the solution form, then prove it by mathematical induction. Three steps: guess, verify base case, inductive step.

Solve T(n) = 2T(n/2) + n, T(1) = 1

Step 1: Guess T(n) = O(n log n)

Assume T(k) ≤ c·k·log k for all k < n

Step 2: Inductive step

T(n) = 2T(n/2) + n ≤ 2·c·(n/2)·log(n/2) + n = cn·(log n - 1) + n = cn log n - cn + n ≤ cn log n if c ≥ 1 ✓

Solve T(n) = T(n-1) + n, T(1) = 1

Step 1: Expand (unroll)

T(n) = T(n-1) + n
= T(n-2) + (n-1) + n
= T(n-3) + (n-2) + (n-1) + n
= T(1) + 2 + 3 + … + n

Step 2: Sum

T(n) = 1 + Σᵢ₌₂ⁿ i = 1 + n(n+1)/2 - 1 = n(n+1)/2 = O(n²)

T(n) = T(n-1) + n → T(n) = Θ(n²)

Recurrence — Tree Method

Recursion Tree Method

Draw the recursion tree: each node = subproblem cost, sum all levels. The total cost = sum of all nodes across all levels.

T(n) = 2T(n/2) + cn

  • Level 0 (root): cost = cn, subproblems = 2×T(n/2)
  • Level 1: 2 nodes × cn/2 each = cn total
  • Level 2: 4 nodes × cn/4 each = cn total
  • Level k: 2ᵏ nodes × cn/2ᵏ each = cn total
  • Depth = log₂n → log₂n + 1 levels, each costs cn

Master Theorem

The Master Theorem

For recurrences of the form T(n) = a·T(n/b) + f(n) where a ≥ 1, b > 1, compare f(n) to n^(log_b a).

T(n) = a·T(n/b) + f(n)
a = subproblems · b = size reduction
f(n) = work outside recursion
Case 1 — leaves dominate

If f(n) = O(n^(log_b(a) - ε)) for some ε > 0, then T(n) = Θ(n^log_b(a))

Case 2 — balanced

If f(n) = Θ(n^log_b(a)), then T(n) = Θ(n^log_b(a) · log n)

Case 3 — root dominates

If f(n) = Ω(n^(log_b(a) + ε)) and regularity condition, then T(n) = Θ(f(n))

Merge Sort: T(n)=2T(n/2)+n

a=2, b=2, f(n)=n. log_b(a)=log₂2=1. f(n)=n=Θ(n¹)=Θ(n^log_b a). Case 2 applies. T(n)=Θ(n log n) ✓

Binary Search: T(n)=T(n/2)+1

a=1, b=2, f(n)=1. log_b(a)=log₂1=0. f(n)=1=Θ(n⁰). Case 2. T(n)=Θ(log n) ✓

Strassen: T(n)=7T(n/2)+n²

a=7, b=2. log₂7≈2.81. f(n)=n²=O(n^(2.81-ε)). Case 1. T(n)=Θ(n^log₂7) ≈ Θ(n^2.81) ✓

T(n)=4T(n/2)+n³

a=4, b=2. log₂4=2. f(n)=n³=Ω(n^(2+1)). Case 3 (check regularity: 4·(n/2)³=n³/2 ≤ cn³ ✓). T(n)=Θ(n³) ✓

Amortised Analysis

Amortised Analysis

Amortised analysis computes the average cost per operation over a sequence of operations — even if some individual operations are expensive, the average may be O(1).

  • Does NOT assume a probability distribution (unlike average-case)
  • The expensive operations happen rarely enough to be "paid for" by the cheap ones
  • Three methods: Aggregate, Accounting (Banker), Potential

Example: Dynamic array — most appends are O(1). Resize (copy all) costs O(n) but happens only every ~n appends. Amortised cost = O(1).

Proof: n appends on a dynamic array

# Capacity doubles on resize:
# 1, 2, 4, 8, 16, … insertions
# Cost of each resize:
# copy 1, 2, 4, 8, … elements
# Total copy work after n appends:
# ≤ 1+2+4+8+…+n = 2n (geometric)
# Total cost: n appends + 2n copies
# = 3n ops = O(n)
# Per append: O(n)/n = O(1) ✓
Other examples

Stack with multipop (aggregate O(1) per op), splay tree (amortised O(log n)), Union-Find with path compression (amortised O(α(n)) ≈ O(1)).

Common Mistakes

Common Complexity Analysis Mistakes

Ignoring the inner loop's range

Assuming inner loop is always O(n) when it actually depends on i. Always substitute and sum: Σᵢ₌₀ⁿ i = n(n+1)/2.

Confusing O with Θ

O(n) is correct for bubble sort worst case — but so is O(n²). Always state the tightest correct bound: Θ(n²) worst, Θ(n) best.

Forgetting input size definition

For graph algorithms: n = vertices, m = edges. O(V+E) is different from O(V²). Always define what n means for your problem.

Adding instead of multiplying loops

Sequential loops add: O(n)+O(n)=O(n). Nested loops multiply: O(n)×O(n)=O(n²). The distinction is critical.

Ignoring hidden function costs

str.join(list) is O(total length). in on a list is O(n). Always check what built-in operations cost.

Confusing time and space

Recursion depth adds to space complexity. A recursive O(n) algorithm may use O(n) stack space even if it "looks" like O(1) space.

Best-case thinking

Reporting O(1) best case for linear search as if it's the algorithm's complexity. Always clarify: "O(1) best, O(n) worst, O(n) average."

Practical Tips

Complexity Analysis — Practical Tips

Step 1: Identify the dominant operation

Find the innermost loop body (the primitive operation that runs most). Count how many times it executes as a function of n.

Step 2: Count loop iterations

For each loop: determine upper/lower bounds on iterations. Substitution or closed-form sum to get f(n). Multiply for nested, add for sequential.

Step 3: Drop constants & lower terms

5n² + 3n + 100 → O(n²). Always simplify to the single dominant term.

Step 4: Check all cases

State best, average, and worst case separately. Use Θ when all cases are equal, O for worst, Ω for best.

Key Takeaways

  • Algorithm analysis measures growth rate — not exact counts. Drop constants and lower-order terms.
  • Big-O is the asymptotic upper bound (worst case). Ω is lower bound. Θ is tight (both).
  • Formal definition: f=O(g) iff ∃c,n₀ such that f(n)≤c·g(n) for all n≥n₀.
  • Complexity hierarchy: O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!).
  • O(n log n) is the optimal lower bound for comparison-based sorting — proven via decision tree argument.
  • Count loop iterations to derive T(n): single loop→O(n), halving loop→O(log n), nested→multiply.
  • Recurrence relations describe recursive algorithms. Solve via substitution, recursion tree, or Master Theorem.
  • Master Theorem: T(n)=aT(n/b)+f(n). Compare f(n) to n^(log_b a) → 3 cases → closed form.
  • Amortised O(1): expensive ops are rare enough that average over sequence is O(1). Example: dynamic array append.
  • Space complexity counts memory use. Distinguish auxiliary space from total. Recursion uses O(depth) stack space.
