Introduction

What Is a Hash Map?

A data structure that maps keys to values using a hash function to compute an array index — enabling O(1) average-case lookup, insert, and delete.

  • Also called: hash table, dictionary, map, associative array
  • Keys must be unique; values can repeat
  • Built on top of a plain array for O(1) index access
  • The hash function converts any key into a valid array index
  • Most widely used data structure in competitive programming & system design

KEY → HASH → INDEX → VALUE

"alice" → 42 → arr[42] → 95

Python dict

d = {"alice": 95, "bob": 87}

Java HashMap

Map<String,Integer> m = new HashMap<>();

Hash Functions

Properties of a Good Hash Function

Deterministic

Same key always produces the same hash value. No randomness at query time.

Uniform Distribution

Hash values spread evenly across all buckets. Minimises clustering and collisions.

Fast to Compute

Hash computation should be O(1) or O(key length) — not a bottleneck.

Avalanche Effect

Small change in key → completely different hash. Prevents patterns in output.

A bad hash function turns O(1) operations into O(n) by putting everything in the same bucket.

# Simple modulo hash for integers
def hash_int(key, size):
  return key % size

# Polynomial hash for strings
def hash_str(s, size):
  h, p = 0, 31
  for c in s:
    h = (h * p + ord(c)) % size
  return h

The Hash Table Structure

The underlying array has a fixed number of buckets (slots). The hash function maps each key to a bucket index.

# Inserting "alice"→95 into size-7 table
key = "alice"
idx = hash(key) % 7 # → 3
table[idx] = ("alice", 95)

# Lookup "alice"
idx = hash("alice") % 7 # → 3
return table[idx][1] # → 95
  • Array size is typically a prime number — reduces clustering
  • Index = hash(key) % capacity

Collision Handling

Collisions Are Inevitable

By the Birthday Paradox, collisions become likely even when the table is only ~70% full — regardless of the hash function's quality.

  • Pigeonhole principle: n keys into m buckets, if n > m, collisions are guaranteed
  • Even with m >> n, probability of first collision reaches 50% at ~√m insertions
  • Two main strategies: Chaining and Open Addressing

Collision Handling

Separate Chaining

Each bucket holds a linked list (or dynamic array) of all key-value pairs that hash to that index.

  • Simple to implement — no reshuffling on collision
  • Degrades to O(n) if all keys land in one bucket
  • Average search: O(1 + α) where α = load factor = n/m
  • Extra memory for pointers — not cache-friendly
  • Used by Java HashMap, Python dict (pre-3.6)
# Chaining insert
def insert(self, key, val):
  i = hash(key) % self.size
  for pair in self.table[i]:
    if pair[0] == key:
      pair[1] = val; return
  self.table[i].append([key, val])

Collision Handling

Open Addressing

All entries stored directly in the array. On collision, probe for the next empty slot using a probing sequence.

  • No extra memory for pointers — cache-friendly
  • Table can fill up — capacity must be > n
  • Deletion is tricky — requires tombstone markers
  • Three probing strategies: Linear, Quadratic, Double Hashing
  • Used by Python dict (3.6+), C++ unordered_map, Robin Hood hashing

Collision Handling

Linear, Quadratic & Double Hashing

Linear Probing

idx = (h + i) % m

  • Simple, cache-friendly
  • Suffers from primary clustering — long runs of filled slots slow probing
# probe sequence
i = 0
while filled:
  idx=(h+i)%m
  i += 1

Quadratic Probing

idx = (h + i²) % m

  • Reduces primary clustering
  • Suffers from secondary clustering — same initial hash → same probe path
# probe sequence
i = 0
while filled:
  idx=(h+i*i)%m
  i += 1

Double Hashing

idx = (h1 + i·h2) % m

  • Eliminates both clustering types
  • Best distribution — requires two hash functions
# probe sequence
h2 = 7-(key%7)
while filled:
  idx=(h1+i*h2)%m
  i += 1

Load Factor

Load Factor & Rehashing

α = n / m

n = entries stored · m = total buckets

  • α < 0.5: Many empty slots — fast but wasteful
  • α ≈ 0.7: Sweet spot for most implementations
  • α > 0.9: Severe degradation — many collisions
  • When α exceeds threshold → rehash: allocate 2× table, re-insert all keys
  • Rehashing is O(n) but amortised O(1) per insert over many operations
def _rehash(self):
  old = self.table
  self.size *= 2 # double capacity
  self.table = [[] for _ in range(self.size)]
  self.count = 0
  for bucket in old:
    for key, val in bucket:
      self.insert(key, val) # re-insert

Java HashMap rehashes at α = 0.75. Python dict rehashes at α = 0.67.

Operations

Insert & Search

INSERT(key, value)

  • Compute index = hash(key) % m
  • If bucket empty → place entry
  • If key exists → update value
  • If collision → resolve (chain or probe)
  • If α > threshold → rehash

SEARCH(key)

  • Compute index = hash(key) % m
  • Scan bucket/probe sequence for key
  • Return value if found, else None
# Full HashMap (chaining)
class HashMap:
  def __init__(self, m=16):
    self.size = m
    self.count = 0
    self.table = [[] for _ in range(m)]

  def _idx(self, key):
    return hash(key) % self.size

  def put(self, key, val):
    i = self._idx(key)
    for pair in self.table[i]:
      if pair[0] == key:
        pair[1] = val; return
    self.table[i].append([key,val])
    self.count += 1

  def get(self, key):
    i = self._idx(key)
    for k, v in self.table[i]:
      if k == key: return v
    return None

Operations

Delete

Deletion in open addressing is tricky — you cannot simply empty the slot or you break the probe chain for other keys.

  • Chaining: Remove entry from the linked list — straightforward O(1) avg
  • Open Addressing: Mark deleted slot with a tombstone sentinel
  • Tombstone slots are skipped during probing but can be reused on insert
  • Too many tombstones → performance degrades → rehash clears them
# Chaining delete
def delete(self, key):
  i = self._idx(key)
  self.table[i] = [
    p for p in self.table[i]
    if p[0] != key
  ]

# Open addressing (tombstone)
DELETED = "__DELETED__"
def delete(self, key):
  i = self._idx(key)
  while self.table[i] is not None:
    if self.table[i][0]==key:
      self.table[i] = DELETED
      return
    i = (i+1) % self.size
Tombstone example

Bucket 2: "bob"→87 deleted → becomes DELETED. Probing past it continues normally.

Worst case O(n) requires a pathological hash function or a deliberate attack. With a good hash function, O(1) is reliable in practice.

Hash DoS Attack

Attackers can craft inputs that all collide, degrading performance to O(n). Defence: use a randomised seed (Python's hash randomisation) or cryptographic hash like SipHash.

defaultdict

Auto-initialises missing keys. from collections import defaultdict
d = defaultdict(list) — appending to missing key creates an empty list.

Counter

Counter(["a","b","a"]) → {"a":2, "b":1}. Counts hashable objects — built on dict.

OrderedDict

Preserves insertion order (Python 3.7+ regular dicts also preserve order). Useful for LRU cache patterns.

Python dict is one of the most optimised data structures ever built — benchmarks often beat hand-rolled C++ maps.

Variants

Hash Sets

A Hash Set is a Hash Map with only keys — no values. Used when you need fast membership testing and deduplication.

  • O(1) average for add, remove, and in operator
  • Supports set operations: union, intersection, difference
  • Python: set() — Java: HashSet<T>
  • Unordered — use TreeSet or sorted set if ordering needed
# Python set operations
A = {1, 2, 3, 4}
B = {3, 4, 5, 6}

# Union → {1,2,3,4,5,6}
A | B

# Intersection → {3, 4}
A & B

# Difference → {1, 2}
A - B

# Membership test — O(1)
3 in A # True

# Deduplication — O(n)
unique = list(set([1,2,2,3]))

Patterns

Common Hash Map Patterns

Frequency Counting

Count occurrences of elements. Used for: anagram detection, top-k elements, majority voting.

Two-Sum Pattern

Store seen values in a map. Check if target - current exists. Reduces O(n²) brute force to O(n).

Grouping / Bucketing

Group items by a computed key. e.g., group anagrams: sort each word → use sorted form as key.

# Two Sum — O(n)
def two_sum(nums, target):
  seen = {}
  for i, n in enumerate(nums):
    diff = target - n
    if diff in seen:
      return [seen[diff], i]
    seen[n] = i

# Group anagrams — O(n·k log k)
from collections import defaultdict
def group_anagrams(words):
  groups = defaultdict(list)
  for w in words:
    key = "".join(sorted(w))
    groups[key].append(w)
  return list(groups.values())

Patterns

LRU Cache — Hash Map + Doubly Linked List

An LRU (Least Recently Used) Cache evicts the least recently accessed item when capacity is full — achievable in O(1) using a HashMap + Doubly Linked List.

  • HashMap: key → node pointer for O(1) lookup
  • Doubly Linked List: maintains access order
  • Move accessed node to head → O(1)
  • Evict tail node when full → O(1)
  • Python: functools.lru_cache or OrderedDict
from collections import OrderedDict

class LRUCache:
  def __init__(self, cap):
    self.cap = cap
    self.cache = OrderedDict()

  def get(self, key):
    if key not in self.cache:
      return -1
    self.cache.move_to_end(key)
    return self.cache[key]

  def put(self, key, val):
    self.cache[key] = val
    self.cache.move_to_end(key)
    if len(self.cache) > self.cap:
      self.cache.popitem(last=False)

Advanced Topics

Consistent Hashing

A technique used in distributed systems so that adding or removing a server only remaps k/n keys on average — not all of them.

  • Map both servers and keys onto a virtual ring (hash ring)
  • Each key is assigned to the nearest server clockwise on the ring
  • Adding a server only takes keys from its immediate neighbour
  • Virtual nodes per server → even load distribution
  • Used in: Amazon Dynamo, Cassandra, CDN routing, Memcached

Key Takeaways

  • A Hash Map maps keys → values using a hash function for O(1) average operations
  • Hash function must be deterministic, fast, and uniformly distributed
  • Collisions are inevitable — handle with Chaining or Open Addressing
  • Chaining: linked list per bucket — simple, handles high load factors
  • Open Addressing: all entries in array — cache-friendly, requires load factor < 1
  • Load factor α = n/m controls performance — rehash when α exceeds threshold
  • Double Hashing gives the best distribution for open addressing
  • Python dict and set are hash map and hash set implementations
  • LRU Cache = HashMap + Doubly Linked List → O(1) all operations
  • Consistent Hashing solves distributed key assignment — used in Dynamo, Cassandra
